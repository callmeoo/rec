# 二手车拍卖车辆详情页优化方案

**制定人：** 产品经理  
**日期：** 2026-01-29  
**目标：** 提升买家对车况的理解度，提高出价率  
**当前问题：** 信息展示不够直观，买家决策困难，出价率低

---

## 📊 一、现状分析

### 1.1 当前详情页问题

```
核心痛点：
1. 车况信息分散，买家需要翻很多页才能看完
2. 检测报告专业术语多，买家看不懂
3. 缺少利润计算工具，买家不知道能赚多少
4. 没有市场行情对比，买家不敢出价
5. 图片质量参差不齐，看不清细节
6. 缺少信任背书，担心买到问题车
```

### 1.2 买家决策路径

```
浏览列表 → 点击详情 → 查看车况 → 计算利润 → 评估风险 → 决定出价
  100%      40%        80%        20%        10%         5%

关键流失点：
- 详情页到查看车况：流失20%（信息太多，找不到重点）
- 查看车况到计算利润：流失60%（没有工具，懒得算）
- 计算利润到评估风险：流失50%（不知道市场行情）
- 评估风险到出价：流失50%（担心风险，不敢出价）
```

---

## 🎯 二、优化目标

### 2.1 核心指标

| 指标 | 当前 | 目标 | 提升幅度 |
|------|------|------|---------|
| 详情页停留时长 | 45秒 | 2分钟 | +166% |
| 详情页到出价转化率 | 5% | 15% | +200% |
| 车况信息查看率 | 30% | 80% | +166% |
| 利润计算器使用率 | 0% | 60% | 新增 |
| 出价犹豫时间 | 2天 | 4小时 | -92% |


### 2.2 设计原则

```
1. 一屏看懂核心信息（3秒决策）
2. 可视化展示车况（图表代替文字）
3. 智能计算利润（自动化工具）
4. 市场行情对比（数据支撑决策）
5. 风险透明化（建立信任）
```

#### 计算逻辑

```javascript
// 利润计算公式
const calculateProfit = (bidPrice, vehicle) => {
  // 1. 成本计算
  const vehicleCost = bidPrice;
  const repairCost = estimateRepairCost(vehicle.conditionScore); // 根据车况评分预估
  const taxFee = bidPrice * 0.02; // 过户税费约2%
  const logisticsCost = calculateLogistics(vehicle.location, buyerLocation);
  const totalCost = vehicleCost + repairCost + taxFee + logisticsCost;
  
  // 2. 售价预估（基于市场数据）
  const marketPrice = getMarketPrice(vehicle.brand, vehicle.series, vehicle.year);
  const estimatedSalePrice = marketPrice.median;
  
  // 3. 利润计算
  const netProfit = estimatedSalePrice - totalCost;
  const profitRate = (netProfit / totalCost) * 100;
  
  // 4. 周转预测
  const turnoverDays = getTurnoverDays(vehicle.modelId);
  const annualizedReturn = (netProfit / totalCost) * (365 / turnoverDays) * 100;
  
  return {
    totalCost,
    estimatedSalePrice,
    netProfit,
    profitRate,
    turnoverDays,
    annualizedReturn
  };
};
```

```javascript
// 相似车源推荐算法
const getSimilarVehicles = (currentVehicle) => {
  const candidates = [];
  
  // 1. 同品牌同车系（年份±1年）
  const sameSeries = vehicles.filter(v => 
    v.brand === currentVehicle.brand &&
    v.series === currentVehicle.series &&
    Math.abs(v.year - currentVehicle.year) <= 1
  );
  candidates.push(...sameSeries.slice(0, 2));
  
  // 2. 同价位区间（±10%）
  const priceRange = currentVehicle.price * 0.1;
  const samePrice = vehicles.filter(v =>
    Math.abs(v.price - currentVehicle.price) <= priceRange &&
    v.id !== currentVehicle.id
  );
  candidates.push(...samePrice.slice(0, 2));
  
  // 3. 同车况等级
  const sameCondition = vehicles.filter(v =>
    v.conditionScore >= currentVehicle.conditionScore - 5 &&
    v.id !== currentVehicle.id
  );
  candidates.push(...sameCondition.slice(0, 2));
  
  // 去重并返回前4个
  return [...new Set(candidates)].slice(0, 4);
};
```

## 📋 四、推荐策略

### 1. 数据筛选和过滤

- 车辆状态：预展中、拍卖中
- 过滤车况评级D和火烧车、泡水车（含"水泡"标签）
- 过滤已登录账号已出价车辆

### 2. 推荐数据

#### 2.1 前端展示

前端展示全平台上拍车辆的15%（按照每日上架不足200台，为双排推荐**固定展示16台车，可配**）

#### 2.2 候选集

#### 候选集 1

**Step 1 — 强匹配（优先）**

- 匹配维度：主打车系 + 主打价格段 + 主打车龄（三维全匹配）
  - 例如买家标签为"大众迈腾 + 3-5万 + 4-6年"，则严格按取该交集下的车源，若满18台则停止向下寻找。
- 若结果 > 目标数量 → 直接使用，反之进入Step 2

**Step 2 — 双维宽泛匹配（解决候选偏差大）**

- 匹配条件：【主打车系】+【价位区差：金额×10%】+【车龄区差：±1年】。
- 动作：将符合该条件的车辆补充进列表（需排除第1层级已有的车，按vid去重）。
- 触发条件：前段1结果 < 16 台。
- 价格容差逻辑：在原价位段的基准值上，允许上下浮动10%的绝对金额，价格不可为负；当最低价格段为"0-3万"时，向下容差计算可出现负数（例如0万的-10%应该是0），价格容差不可为负。
  - 举例：如果买家标签是3-5万，允许推荐起拍价在2.7万（3万-10%）至5.5万（5万+10%）之间的同车系，这样能扩充车源，让候选集能接近2.98万的车源也进入匹配范围。
- 车龄容差逻辑：在买家主打车龄标签的上下限上，各放宽1年。
  - 举例：若买家主打车龄为4-6年，"容差"扩为3-7年。若买家主打车龄为2年以内（即0-2年）：向下不可为负，向上放宽年，容差扩为0-3年。若买家主打车龄为10年以上：向下放宽，容差扩为9年以上。
- 若结果 > 目标数量 → 直接使用，反之进入Step 3

**Step 3 — 放开车系（一级扩展）**

- 匹配维度：主打价格段 + 主打车龄（去掉车系限制）
- 若结果 > 目标数量 → 使用，反之进入Step 4

**Step 4 — 兜底（二级扩展）**

- 匹配维度：无约束，从候选集2（平台置顶TOP50）中补充剩余缺口
- 若候选集2仍不足，则从候选集3（全站兜底）补至目标数量
- 去重机制：在经历阶段1到阶段4的漏斗过滤时，拉取的数据集可能会有重合，需做去重处理，避免同一台车在候选展示两次。

#### 候选集 2（平台置顶）

- 全站置顶分TOP 50车系，同车系占比 < 40%；候选集内的车辆需要差异大于60；

#### 候选集 3（全站兜底）

- 若以上候选结果集都不足，则余车辆推荐位用"全站热销兜底"。
- 全站热销：在拍车辆按照出价次数从高到低排序，出价次数一致，按余价格结束时间升序排列；若拍卖结束时间和出价次数都是相同，则采随机排序打散。
- 在全局过滤规则外，候选集3排除与前出价次数为0的车辆

### 2.3 排序规则

- 同一个候选集：按【拍卖结束时间(升序)】，拍卖结束时间相同，按出价次数(降序)；若拍卖结束时间和出价次数相同，随机排序。
- 打散策略：前10位中，同一品牌连续 ≤3 台，同一车系连续 ≤3 台。

### 2.4 异常处理

- 为保持系统高可用，当推荐服务出现故障，接口超时未返回，则直接调用**缓存**中的上一次计算结果。
- 在该取缓存列表后，将状态不符（已拍卖结束的）车辆从缓存中过滤，若数量不足，则用候选集3（全站热销兜底）进行补充。
- **三个候选集集合并后仍不足16台时，保持实际数量展示，不强制补满**

### 2.5 用户未登录情况下的推荐策略

- 取：候选集2 & 候选集3 数据

```javascript
const fetchWithRetry = async (url, retries = 3) => {
  for (let i = 0; i < retries; i++) {
    try {
      return await fetch(url);
    } catch (error) {
      if (i === retries - 1) throw error;
      await new Promise(resolve => setTimeout(resolve, 1000 * (i + 1)));
    }
  }
};
```
