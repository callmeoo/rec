# 友盟埋点集成说明

## 概述

推荐系统已集成友盟统计埋点，用于追踪用户行为和推荐效果。所有埋点数据可在后台的"数据看板"模块查看。

**重要提示：** 友盟数据不是实时数据，有1天延迟。数据看板展示的是历史统计数据，不适合用于实时监控。

## 数据查看

### 后台数据看板
**访问路径：** 推荐管理 > 数据看板

**数据特性：**
- 非实时数据，有1天延迟
- 支持查看近7天、30天、90天的历史数据
- 数据按日期维度聚合展示
- 仅统计移动端数据（PC端流量较低，不纳入统计）

**功能模块：**
1. **核心指标卡片**
   - 总曝光量
   - 总点击量
   - 点击率(CTR)
   - 出价率
   - 平均出价

2. **趋势分析图表**
   - 7天/30天/90天数据趋势
   - 曝光量、点击量、出价量趋势

3. **候选集命中分布（移动端）**
   - 画像匹配占比
   - 平台周转占比
   - 全站兜底占比
   
   **总曝光次数定义：** 推荐模块中所有车辆的曝光次数总和（即用户看到推荐列表的次数）
   
   **计算公式：**
   - 画像匹配占比 = (画像匹配推荐车辆曝光次数 / 总曝光次数) × 100%
   - 平台周转占比 = (平台周转推荐车辆曝光次数 / 总曝光次数) × 100%
   - 全站兜底占比 = (兜底推荐车辆曝光次数 / 总曝光次数) × 100%
   
   **计算示例：**
   - 用户A打开推荐页面，看到10辆车（3辆画像匹配 + 5辆平台周转 + 2辆兜底），则产生10次曝光
   - 如果当天共有1000次这样的曝光，其中画像匹配车辆被曝光450次
   - 则画像匹配占比 = 450 / 1000 = 45%
   
   **数据来源：** 每个推荐车辆在埋点时会标记其候选集类型（candidateSet字段：1=画像匹配，2=平台周转，3=全站兜底），统计时按候选集类型分组计算占比。

4. **推荐位效果分析（移动端）**
   - 1-3位效果
   - 4-6位效果
   - 7-10位效果
   - 10位后效果

## 友盟SDK集成步骤

### 1. 安装友盟SDK
```bash
npm install umeng-web-analytics
```

### 2. 配置友盟AppKey
在 `src/utils/umengTracking.js` 中修改：
```javascript
const UMENG_APP_KEY = 'YOUR_UMENG_APP_KEY'; // 替换为实际的友盟AppKey
```

### 3. 初始化友盟
在 `src/main.jsx` 或 `src/App.jsx` 中调用：
```javascript
import { initUmeng } from './utils/umengTracking';

// 在应用启动时初始化
initUmeng();
```

### 4. 实现sendToUmeng函数
在 `src/utils/umengTracking.js` 中完善 `sendToUmeng` 函数，调用友盟SDK的实际API：
```javascript
const sendToUmeng = (eventData) => {
  // 调用友盟SDK
  if (window.aplus_queue) {
    window.aplus_queue.push({
      action: 'aplus.record',
      arguments: [eventData.event_name, 'CLK', eventData]
    });
  }
};
```

## 注意事项

1. **用户ID获取**：当前使用模拟数据，实际使用时需要从用户登录状态中获取真实的用户ID
2. **候选集类型**：需要在推荐引擎中为每个车辆添加 `candidateSet` 字段
3. **推荐位位置**：需要在推荐列表中为每个车辆添加 `position` 字段（1-based索引）
4. **数据隐私**：确保埋点数据符合隐私政策和数据保护法规
5. **性能优化**：埋点调用应该是异步的，不应阻塞主线程

## 后续优化建议

1. **批量上报**：将多个埋点事件批量上报，减少网络请求
2. **离线缓存**：网络异常时将埋点数据缓存到本地，恢复后上报
3. **采样率控制**：对于高频事件可以设置采样率，减少数据量
4. **A/B测试**：集成A/B测试功能，对比不同推荐策略的效果
5. **实时监控**：添加实时数据监控和告警功能
