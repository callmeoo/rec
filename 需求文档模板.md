# 需求文档模板

## 📋 如何编写需求文档

这是一份标准的产品需求文档（PRD）模板，帮助你清晰地向开发团队传达需求。

---

## 1️⃣ 基本信息

```markdown
# [功能名称]

**需求提出人：** 张三  
**需求日期：** 2026-01-26  
**优先级：** 高/中/低  
**预期上线时间：** 2026-02-15  
**相关产品：** 二手车拍卖平台
```

---

## 2️⃣ 需求背景

**说明为什么要做这个功能**

```markdown
## 需求背景

当前平台用户在浏览车辆时，缺少个性化推荐功能，导致：
- 用户需要手动搜索大量车辆
- 转化率较低（当前转化率5%）
- 用户停留时间短（平均3分钟）

竞品分析：
- 瓜子二手车有"猜你喜欢"功能，用户停留时间提升40%
- 优信二手车的推荐系统带来30%的成交转化

预期收益：
- 提升用户停留时间至5分钟
- 提升转化率至8%
- 增加用户粘性
```

---

## 3️⃣ 目标用户

**明确这个功能是给谁用的**

```markdown
## 目标用户

### 主要用户群体
1. **购车意向用户**
   - 年龄：25-45岁
   - 特征：有明确购车需求，浏览过多辆车
   - 痛点：信息过载，不知道选哪辆

2. **观望用户**
   - 特征：偶尔浏览，没有明确目标
   - 痛点：缺少引导，容易流失

### 使用场景
- 场景1：用户浏览完某辆车后，想看相似车辆
- 场景2：用户打开APP首页，想快速找到感兴趣的车
- 场景3：用户收藏了几辆车，想看更多同类推荐
```

---

## 4️⃣ 功能需求

**详细描述功能点**

```markdown
## 功能需求

### 4.1 核心功能

#### 功能1：智能推荐列表
**描述：** 根据用户行为推荐车辆

**输入：**
- 用户浏览历史
- 用户收藏记录
- 用户搜索关键词

**输出：**
- 推荐车辆列表（每页20辆）
- 推荐理由标签

**规则：**
1. 优先推荐用户浏览过的品牌
2. 价格区间与收藏车辆相近（±10万）
3. 正在拍卖的车辆优先展示
4. 每次"换一批"展示不同车辆

**示例：**
用户浏览过：奔驰E300、宝马5系
推荐：奥迪A6、特斯拉Model S（同价位豪华车）

---

#### 功能2：多维度筛选
**描述：** 用户可以按条件筛选推荐结果

**筛选维度：**
- 价格区间：0-100万（滑块选择）
- 品牌：多选（奔驰、宝马、奥迪等）
- 车龄：2015-2024年（多选）
- 里程：0-30万公里（滑块选择）

**交互：**
1. 点击"筛选"按钮打开侧边栏
2. 选择条件后点击"应用"
3. 列表实时更新
4. 支持"重置"清空所有条件

---

#### 功能3：收藏功能
**描述：** 用户可以收藏喜欢的车辆

**交互：**
- 点击卡片右上角❤️图标收藏
- 已收藏显示红色实心❤️
- 再次点击取消收藏

**数据记录：**
- 记录收藏时间
- 用于优化推荐算法

---

### 4.2 辅助功能

#### 功能4：换一批
**描述：** 刷新推荐列表

**规则：**
- 保持筛选条件不变
- 展示不同的推荐车辆
- 避免重复展示已浏览车辆

#### 功能5：推荐理由
**描述：** 告诉用户为什么推荐这辆车

**理由类型：**
- "您浏览过奔驰品牌"
- "与您收藏的车辆相似"
- "热门拍卖中"
- "低里程精品车"
- "准新车推荐"
```

## 6️⃣ 交互说明

**描述用户操作和系统反馈**

```markdown
## 交互说明

### 6.1 页面加载
1. 进入页面显示加载动画（骨架屏）
2. 0.8秒后展示推荐列表
3. 默认展示20辆车

### 6.2 筛选操作
1. 点击"筛选"按钮
2. 从右侧滑出筛选面板（300ms动画）
3. 选择条件
4. 点击"应用"
5. 面板关闭，列表更新（显示加载动画）

### 6.3 收藏操作
1. 点击❤️图标
2. 图标变红色（0.2s动画）
3. 显示提示"已收藏"（2秒后消失）
4. 数据同步到后端

### 6.4 查看详情
1. 点击车辆卡片任意位置（除收藏按钮）
2. 跳转到车辆详情页
3. 记录浏览行为

### 6.5 换一批
1. 点击"换一批"按钮
2. 按钮显示加载图标（旋转动画）
3. 0.8秒后列表更新
4. 滚动到页面顶部
```

## 📝 需求文档编写建议

### ✅ 好的需求文档特点
1. **清晰明确**：避免模糊词汇（"可能"、"大概"）
2. **有理有据**：说明为什么要做
3. **可量化**：有明确的指标和标准
4. **完整性**：覆盖所有场景和边界情况
5. **可执行**：开发能直接根据文档实现

### ❌ 避免的问题
1. 需求描述过于简单："做一个推荐功能"
2. 缺少业务背景和目标
3. 没有明确的验收标准
4. 遗漏异常情况处理
5. 技术术语过多，产品和开发理解不一致

### 📋 推荐格式
- **Markdown**：易编辑、易版本管理
- **Word/PDF**：正式文档
- **Notion/飞书文档**：协作编辑
- **Axure/Figma**：带原型的需求
