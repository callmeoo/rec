# BMS后台 - 推荐算法参数配置模块需求文档

## 📋 文档信息

**文档版本**：V2.0.0  
**创建日期**：2026-01-22  
**所属模块**：BMS → 推荐管理 → 参数配置  
**优先级**：P1（核心功能）  

---

## 一、需求概述

### 1.1 功能定位

为运营和产品人员提供推荐算法的参数配置界面，支持灵活调整推荐策略，**无需发布代码**即可优化推荐效果。

### 1.2 核心原则

✅ **纯后台配置**：所有参数都在BMS管理，代码只提供初始默认值  
✅ **动态推荐数量**：支持16-40台动态调整，根据库存和用户行为自适应  
✅ **分类生效**：轻量级配置立即生效，重计算配置次日0点生效  
✅ **无审批流**：编辑后直接生效，简化流程  
✅ **模拟计算**：修改前可预览影响，降低风险  

### 1.3 功能入口

**路径**：BMS后台 → 推荐管理 → 参数配置

**权限**：
- 产品经理：可查看、可编辑
- 运营管理员：可查看、可编辑
- 系统管理员：可查看、可编辑、可查看变更记录
- 数据分析师：仅可查看
- 普通用户：无权限

## 二、配置项设计

### 2.1 配置项总览

| 参数名称 | 默认值 | 取值范围 | 配置类型 | 生效时机 | 影响模块 |
|:---------|:------|:---------|:---------|:---------|:---------|
| 二进宫判定周期 | 30天 | 14~90天 | 整数输入框 | **次日0点** | 周转分计算 |
| 周转排行统计周期 | 60天 | 60/90/120/180 | 下拉单选 | **次日0点** | 周转分排行 |
| 兜底历史天数 | 7天 | 5~30天 | 整数输入框 | **次日0点** | 候选集3 |
| 周转分权重 | 成交率0.7<br>耗时0.3 | 各0~1<br>和为1 | 双滑块（互斥） | **次日0点** | 周转分计算 |
| 用户画像统计周期 | 60天 | 60/90/120/180 | 下拉单选 | **次日0点** | 买家画像 |
| 样本充足判定 | 上架≥6<br>成交≥3 | 上架3~20<br>成交2~10 | 双整数输入框 | **次日0点** | 周转分计算 |
| 推荐数量-最小值 | 16台 | 15~30 | 整数输入框 | **立即生效** | 推荐位数量 |
| 推荐数量-最大值 | 40台 | 30~60 | 整数输入框 | **立即生效** | 推荐位数量 |
| 库存覆盖率 | 15% | 10~20% | 滑块 | **次日0点** | 推荐位数量 |

**说明**：
- **次日0点生效**：需要重跑离线任务（如画像、周转分），定时任务触发
- **立即生效**：仅更新缓存/配置，下次推荐请求即生效

#### 配置项1️⃣：二进宫判定周期

**参数键名**：`second_listing_interval_days`

**功能说明**：判定车辆是否为"二进宫"（重新上架）的时间间隔阈值

**业务逻辑**：
```
如果 (本次上架时间 - 上次结束时间) > 该配置值:
    判定为"二进宫"
    重置成交耗时计算周期，忽略之前的历史
否则:
    视为连续拍卖，累计计算成交耗时
```

**配置界面**：
```
┌────────────────────────────────────────┐
│ 二进宫判定周期                          │
├────────────────────────────────────────┤
│ 当前值：[ 30 ] 天                      │
│          ↑ 可输入，范围14~90           │
│                                        │
│ 说明：车辆距离上次撤拍/流拍超过该天数， │
│      视为重新入库，成交耗时重新计算     │
│                                        │
│ 生效时机：次日0点生效               │
└────────────────────────────────────────┘
```

**验证规则**：
- 必须为整数
- 范围：5 ≤ 值 ≤ 30
- 实时校验，超出范围时红色提示


#### 配置项3️⃣：周转分权重

**参数键名**：`turnover_score_weights`

**功能说明**：周转分计算公式中成交率和耗时的权重配比

**业务逻辑**：
```
周转分 = 成交率评分 × 权重A + 耗时评分 × 权重B
其中：权重A + 权重B = 1.0
```

**配置界面**：
```
┌────────────────────────────────────────┐
│ 周转分权重配置                          │
├────────────────────────────────────────┤
│ 成交率权重：[ 0.7 ]                    │
│ ←━━━━━━━━●━━━━━━━━━━━━━━━━━→         │
│ 0.0              0.5              1.0  │
│                                        │
│ 耗时权重：[ 0.3 ] (自动计算)           │
│ ←━━━━●━━━━━━━━━━━━━━━━━━━━━→         │
│ 0.0              0.5              1.0  │
│                                        │
│ 当前公式：                              │
│ 周转分 = 成交率评分×0.7 + 耗时评分×0.3 │
│                                        │
│ 💡 提示：                              │
│ • 成交率权重越高，越倾向"能卖出"的车型  │
│ • 耗时权重越高，越倾向"卖得快"的车型    │
│                                        │
│ 生效时机：次日0点生效               │
│                                        │
│ [ 模拟计算TOP100车型变化 ]             │
└────────────────────────────────────────┘
```

**交互逻辑**：
- 拖动成交率权重滑块，耗时权重自动调整为 `1 - 成交率权重`
- 显示实时公式预览
- 点击"模拟计算"按钮，预览TOP100车型排名变化

**验证规则**：
- 成交率权重 + 耗时权重 = 1.0（强制互斥）
- 步长：0.1
- 范围：各0.1~0.9

#### 配置项5️⃣：样本充足判定

**参数键名**：`sample_sufficient_threshold`

**功能说明**：车型周转分计算的最小样本量要求

**业务逻辑**：
```
if (某车型上架数 >= 上架数下限) AND (某车型成交数 >= 成交数下限):
    样本充足，参与周转分计算
else:
    样本不足，不参与周转分，标记为"数据不足"
```

**配置界面**：
```

│ 样本充足判定标准                        │
├────────────────────────────────────────┤

│ 成交数下限：[ 3 ] 台  (范围2~10)       │
│                                        │
│ ✅ 满足以上两个条件的车型才参与周转分计算│
│                                        │
│ 💡 提示：                              │
│ • 阈值过高 → 有效车型减少              │
│ • 阈值过低 → 数据不稳定                │
│                                        │
│ 生效时机：次日0点生效               │
│                                        │
│ [ 模拟计算样本充足车型数变化 ]         │
└────────────────────────────────────────┘
```

**验证规则**：
- 成交数下限 ≤ 上架数下限
- 上架数：3 ≤ 值 ≤ 20
- 成交数：2 ≤ 值 ≤ 10
- 必须为整数

#### 配置项6️⃣-8️⃣：推荐数量动态调整参数

**功能说明**：控制首页推荐位数量的动态调整范围

**业务逻辑**：
```
基础推荐位数量 = 当前库存 × 库存覆盖率
最终推荐位数量 = max(最小值, min(最大值, 基础推荐位数量))
```

**配置界面**：
```
┌────────────────────────────────────────┐
│ 推荐数量动态调整配置                    │
├────────────────────────────────────────┤
│ 最小推荐位数量：[ 16 ] 台 (范围15~30)  │
│ 最大推荐位数量：[ 40 ] 台 (范围30~60)  │
│                                        │
│ 库存覆盖率：[ 15% ]                    │
│ ←━━━━━━━●━━━━━━━━━→                 │
│ 10%            15%            20%      │
│                                        │
│ 💡 计算示例（当前库存200辆）：          │
│ 基础数量 = 200 × 15% = 30台            │
│ 最终数量 = max(16, min(40, 30)) = 30台│
│                                        │
│ 生效时机：次日0点生效                │
│ （需重跑离线任务，定时任务触发）        │
└────────────────────────────────────────┘
```

**验证规则**：
- 最小值 < 最大值
- 最小值：15 ≤ 值 ≤ 30
- 最大值：30 ≤ 值 ≤ 60
- 库存覆盖率：10% ≤ 值 ≤ 20%，步长1%

## 三、生效时机设计

### 3.1 分类规则

| 生效类型 | 适用配置项 | 生效流程 | 时效性 |
|:---------|:----------|:---------|:------|
| **次日0点生效** | • 二进宫判定周期<br>• 周转排行统计周期<br>• 兜底历史天数<br>• 周转分权重<br>• 画像统计周期<br>• 样本充足判定<br>• 库存覆盖率 | 保存→标记"待生效"→次日0点定时任务→状态变"已生效" | 12-24小时 |
| **立即生效** | • 推荐数量最小值<br>• 推荐数量最大值 | 保存→更新Redis缓存→下次推荐请求生效 | 秒级 |

### 3.2 次日0点生效流程

```
用户修改配置
    ↓
点击"保存并生效"
    ↓
后端保存到数据库
    ↓
状态标记为"待生效"
    ↓
记录变更日志
    ↓
前端提示："保存成功，将于明日0点生效"
    ↓
次日凌晨0点，定时任务触发
    ↓
读取"待生效"状态的配置
    ↓
执行对应的重算逻辑：
    - 重算周转分（如果权重改了）
    - 重算画像（如果统计周期改了）
    ↓
更新状态为"已生效"
    ↓
清除相关缓存
    ↓
配置正式生效
```

### 3.3 立即生效流程

```
用户修改配置
    ↓
点击"保存并生效"
    ↓
后端保存到数据库
    ↓
更新Redis缓存
    ↓
状态直接标记为"已生效"
    ↓
记录变更日志
    ↓
前端提示："保存成功，已立即生效"
    ↓
下次推荐请求读取新配置
    ↓
配置已生效
```

```python
    """
    读取配置的优先级

    # 1. 优先从数据库读取
    config = db.query(RecommendationConfig).filter_by(
        config_key=config_key,

    if config:
        return config.config_value
    
    # 2. 数据库无配置，使用代码默认值
```

### 9.2 次日0点生效的定时任务

```python
# 定时任务：每日0点执行
@schedule.cron('0 0 * * *')
def apply_pending_configs():
    """
    应用待生效的配置
    """
    # 1. 查询所有待生效的配置
    pending_configs = db.query(RecommendationConfig).filter_by(
        status='pending'
    ).all()
    
    for config in pending_configs:
        # 2. 根据配置类型，执行对应的重算逻辑
        if config.config_key == 'turnover_score_weights':
            # 重算周转分
            recalculate_turnover_scores(config.config_value)
        
        elif config.config_key == 'user_profile_stat_days':
            # 重算画像
            recalculate_user_profiles(config.config_value)
        
        # ...其他配置
        
        # 3. 更新状态为已生效
        config.status = 'effective'
        config.effective_time = datetime.now()
        db.commit()
        
        # 4. 清除相关缓存
        clear_related_cache(config.config_key)
```

### 9.3 立即生效的缓存更新

```python
def update_immediate_config(config_key, config_value):
    """
    更新立即生效的配置
    """
    # 1. 保存到数据库
    db.query(RecommendationConfig).filter_by(
        config_key=config_key
    ).update({
        'config_value': config_value,
        'status': 'effective',
        'effective_time': datetime.now()
    })
    db.commit()
    
    # 2. 更新Redis缓存
    redis.set(f'config:{config_key}', config_value)
    
    # 3. 清除推荐结果缓存（如果影响推荐）
    if config_key in ['recommendation_count_min', 'recommendation_count_max']:
        redis.delete('recommendation:*')
```
